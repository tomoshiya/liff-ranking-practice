# **1.** **アプリ概要**

RankQuestは、「自分のランキングを他の人が当てる」シンプルなルールのコミュニケーションゲームです。参加者全員がお題に対して自分のランキングを入力し、全員が互いのランキングを予想します。正確に当てるほど高得点になる仕組みで、ゲームを通じて「相手が何を大切にしているか」を楽しく知ることができます。

_コアバリュー：話題が広がる！仲が深まる！_

_キャッチコピー：「ランクを読み合うコミュニケーションゲーム」_

# **2.** **ターゲットユーザー**

# **3.** **ゲームの仕組み**

## **3-1.** **基本フロー**

•      ① テーマを決める（アプリ内テーマ一覧から選ぶ or オリジナル入力）

•      ② 参加者全員が自分のランキングを入力（TOP5・最大50文字/項目）

•      ③ 参加者全員が互いのランキングを予想

•      ④ 結果発表・スコア確認（演出あり）

•      ⑤ 盛り上がったら再戦・テーマ変更

※ 参加者全員が出題者かつ回答者。参加人数が増えるほどスコアの上限も増加します。

## **3-2.** **スコアリング**

|   |   |   |
|---|---|---|
|**判定**|**条件**|**得点**|
|あたり|完全一致（同じ順位）|+2pt|
|おしい|±1位以内|+1pt|
|はずれ|それ以外|+0pt|

最大得点：10pt / 人（5ランク × 2pt）×（参加人数 − 1）

例：8人参加時の最大得点 = 10pt × 7人 = 70pt

※ スコアロジックは残論点。「した率 × され率」の掛け算案なども検討中。

## **3-3. 2****つの評価軸**

結果発表では以下2軸でスコアを分解して表示します（表現は仮称）。

|   |   |   |
|---|---|---|
|**軸**|**仮称**|**意味**|
|正解した率|（仮）ヨミ力・理解力|自分が他の人のランキングをどれだけ当てたか|
|正解され率|（仮）オープン度・開示力|自分のランキングが他の人にどれだけ当てられたか|
|ズレ幅|した率 − され率|した率が高い＝ミステリアス、され率が高い＝わかりやすい|

※ 軸の表現（ヨミ力・オープン度等）はβ版中に検討・確定予定。

# **4.** **ゲームモード**

|   |   |   |   |
|---|---|---|---|
|**モード**|**概要**|**人数**|**同期方式**|
|みんなでランクエ（multi）|各自のスマホから参加。部屋番号で入室。|3〜10人程度|Firebase Realtime DB|
|ふたりでランクエ（beta）|2人がそれぞれのスマホでプレイ。|2人|Firebase Realtime DB|
|端末1つでランクエ（local）|1台のスマホを回しながらプレイ。|2人以上|Firebase不使用|

# **11.** **テーマパック設計**

## **11-1.** **テーマの役割**

Rank-Qにおいてテーマ（お題）は、ゲーム体験の質を左右する最重要要素のひとつです。テーマ設計においては、スコアを競うこと以上に「答えた後に会話が生まれるか」を最優先とします。

## **11-2.** **テーマの定型ルール**

テーマテキストは「〜〜TOP5」で統一します。将来的にTOP3など可変対応が必要な場合は、アプリ側で動的に数値を差し替える設計とします。

## **11-3.** **良いテーマの基準**

•      選びたくなる問いであること：一覧で見たときに「これやりたい！」と感じる引力があるか

•      答えに個性が出ること：誰が答えても同じになるテーマはNG

•      答えた後に会話が生まれること：「えっそれなんで？」「わかる〜」が起きるテーマ

•      前段で引きつけること：「TOP5」の前の言葉に熱量・具体性・毒や愛があること

## **11-4.** **テーマ管理方法**

テーマはGoogleスプレッドシートで管理し、Google Apps Script（GAS）経由でFirebaseにインポートします。

列構成：パックID / パック名（日）/ パック名（英）/ テーマID / テーマテキスト / ステータス / お気に入り / 難易度 / トーン / メモ / 追加日

•      currentSession：進行中のゲームの一時データ

**正式版（Firebase）**

•      全ゲーム履歴・回答内容・予想内容を永続保存

•      閲覧権限をサブスクで差別化（データは全ユーザー分保存・表示範囲のみ制限）

## **13-3. LocalStorage設計確定（2026年3月）**

### キー構成

### 設計方針
- 他者のTOP5も保存する（ゲームで共有された情報・LocalStorage端末内保存はリスク低）

### データ保存方式の整理

| 方式 | 揮発性 | β版での用途 |
|------|--------|------------|

| SessionStorage | タブを閉じると消える | 使用しない |
| **LocalStorage** | **端末に永続保存** | **履歴・設定・セッション復元** |
| Firebase | サーバー永続保存 | リアルタイム同期（既存）・将来の全履歴保存 |
